# 不要裁判，多派几个选手：一种基于 AST 归一化的分布式 AI 验证协议

当你的 AI 写出了看起来正确的错误代码，你需要另一台 AI 来审查它——但谁来审查审查者？

## 一个让人不安的事实

GPT-4o 写代码的正确率在 HumanEval 上已经突破 90%。这意味着你让它写 100 段代码，有 10 段是有 bug 的，而且你看不出来。

当代码审查也是 AI 做的时候，整个信任链条就变成了一个递归问题：

```
AI 写代码 → AI 审查 → AI 修复 → AI 再审 → ...
```

一个模型在验证自己的输出。而所有自回归语言模型都有一个天然缺陷：它不知道自己不知道。

统计学意义上的"最可能的下一个 token"和"逻辑上正确的下一行代码"是两回事。

## 现有方案为什么不够好

**LLM-as-a-Judge：** 用 GPT-4 审 GPT-4。问题在于 GPT-4 也会犯错，只是错误率略低，成本翻倍。递归递归，没有尽头。

**Self-Consistency（多数投票）：** 同一个 prompt 采样 N 次，选高频答案。但 LLM 的错误不是随机的——同一个训练集导致的系统性偏见，投票投不出去。

**CI/CD：** 告诉你"没有语法错误"，但检测不了"能编译但逻辑不对"的代码。一个有 bug 的二分查找可以通过编译器，但跑测试会死循环。

**共同问题：** 都需要一个更高层级的"裁判"。裁判本身不可信，裁判不可信谁来裁判裁判？这是个递归问题，递归的尽头要么是无限嵌套，要么是一个绝对权威——而 AI 领域没有绝对权威。

## 换个思路：不要裁判，多派几个选手

假设你有三台机器，每台都跑了同一个 200M 的小模型。给它们同一个 prompt——用 Go 写一个函数，接收两个整数，返回它们的和。

三台机器各自独立推理。如果它们的输出完全一致——**不是"意思差不多"，字符串级别的完全一致**——你有 99% 的把握这个结果是正确的。

因为三台机器独立犯错且犯一模一样错误的概率，远低于单台机器犯错。

但这里有一个坑：模型推理有随机性。同样的模型、同样的 prompt，由于 seed 不同、浮点精度差异、量化方式不同，两次推理可能**文字不同但逻辑等价**：

```
机器 A 输出：func add(x, y int) int { return x + y }
机器 B 输出：func sum(a, b int) int { return a + b }
```

你需要一种方法把这两段代码归一化成同一种表示，然后再做哈希比对。这就是 AST 归一化做的事。

## AST 归一化（附代码）

### Level 0：原始代码
```
func add(x, y int) int { return x + y }
```

### Level 1：剥离表面差异
移除注释、空白、换行、缩进。正常化引号和括号位置。这段代码在多语言和多环境之间，肉眼看起来可能不同，但机器的视角应该忽略这些。

### Level 2：变量名置换（核心）
核心算法就十几行：

```python
class ASTNormalizer:
    def __init__(self, level=2):
        self.level = level
        self._var_map = {}
        self._var_counter = 0
    
    def _get_rename(self, name):
        if name not in self._var_map:
            self._var_map[name] = f"__VAR_{self._counter}__"
            self._counter += 1
        return self._var_map[name]
```

所有变量名、函数名、参数名替换为统一的占位符。上面的代码归一化后都变成：

```
func __VAR_0__(__VAR_1__, __VAR_2__ int) int { return __VAR_1__ + __VAR_2__ }
```

不管原始代码叫 `add(x, y)` 还是 `sum(a, b)` 还是 `compute(n1, n2)`，归一化后结构完全一致。

### Level 3：AST 结构归一化（Python）

对于 Python，直接用内置 ast 模块做 AST 级别转换：

```python
import ast

class PythonNormalizer(ast.NodeTransformer):
    def visit_Name(self, node):
        if isinstance(node.ctx, ast.Store):
            node.id = self._get_rename(node.id)
        elif isinstance(node.ctx, ast.Load):
            if node.id in self._var_map:
                node.id = self._var_map[node.id]
        return node
    
    def visit_arg(self, node):
        if node.arg not in dir(__builtins__):
            node.arg = self._get_rename(node.arg)
        return node
```

Go 三种语言（Go 用正则、Python 用 ast、TypeScript 用字符串解析）都走通后，统一计算 SHA-256：

```python
import hashlib

normalized = normalizer.normalize(code, language)
hash_value = hashlib.sha256(normalized.encode()).hexdigest()
```

## 验证协议

```
Step 1: 母模型接收用户请求
Step 2: 通过 BT 协议分发到 N 个工作节点
Step 3: 每个节点独立运行推理，产出代码
Step 4: 节点对输出做 AST 归一化 → SHA-256
Step 5: 哈希回传母模型
Step 6: M of N 哈希一致 → 共识达成
Step 7: 不一致的节点 → 标记 + 返回原始代码对比
```

设计中刻意避开了几个陷阱：

**不用 AI 做裁判：** 共识完全基于确定性哈希比对，没有任何模型参与判断。不是"AI 审核 AI"，是"哈希比对哈希"。

**M-of-N 可配置：** N=3 时要求 3 个全匹配，N=5 时可要求 4/5 匹配，N=1 退化为单机执行（信任该节点）。**这套协议的强度不在于单个节点的可靠性，在于共识门限。**

**恶意节点不惩罚，只标记：** 哈希不一致只意味着"这个节点和其他节点不同"，不一定是有意作恶。可能是量化误差、温度设置、甚至硬件 bug。但不管什么原因，不一致的结果不被接受。

## 实测数据

在 Go、Python、TypeScript 三种语言上做了完整验证：

| 测试场景 | 归一化后哈希一致？ |
|---------|-----------------|
| 同一代码，格式不同 | ✅ 一致 |
| 同一逻辑，变量名不同 | ✅ 一致 |
| 同一逻辑，函数名/参数名全不同 | ✅ 一致 |
| 逻辑不同（恶意节点） | ✅ 不一致，被检出 |
| 多文件跨函数 | ✅ 一致 |

三台机器独立运行同一个 200M 模型，相同 prompt，**即使代码表面写法不同（变量名、格式化、注释风格），归一化后的哈希完全一致。** 恶意节点篡改逻辑，哈希立即不一致，被标记。

## 这个架构意味着什么

### 算力经济的结构变化

现有分布式推理方案（Petals、Exo）的思路是多台机器拼起来跑一个大模型。这个方案反过来走：**每台机器跑一个小模型，通过冗余执行而不是模型分片来获得可靠性。**

消费级 GPU（GTX 1060，4GB 显存）跑 200M 模型的推理延迟约 200ms。三台 1060 加起来没有一张 A100 的零头，但能提供可验证的结果。

**你不必信任硬件。你信任共识。**

### 对 AI 服务定价的启示

- 传统定价：按 token 计费，算你用了多少 GPU 时间
- 这里的定价：按共识次数计费，算你得到了多少次独立验证

后者的边际成本有趋近于零的可能——因为闲置节点的算力对社会来说是免费资源。一万台老旧笔记本 = 一个免费的数据中心。

### 对幻觉防御的路径转移

所有基于"模型自我纠正"的幻觉防御方案，本质上是在试图让一个会说谎的人更诚实。这个方案走的是另一条路：

> 我不关心你诚不诚实，我只关心你和隔壁节点说的能不能对上。

## 抛几个开放问题

1. **冗余执行浪费算力吗？** 是的。但浪费算力的代价和"推理结果不可靠"的代价比，哪个更大？对金融交易代码、医疗诊断代码来说，浪费 3 倍算力换来确定性验证，是划算的。

2. **只在代码领域有效？** 当前是的。但对 SQL 生成、JSON Schema 输出、配置生成这类结构化输出场景，同样思路可以用。自然语言呢？语义等价性的确定性验证目前还是开放问题。

3. **P2P 网络的冷启动怎么解决？** 没人就没节点，没节点就没法验证。初期需要种子节点，或者走混合模式——跑不通 P2P 时退化为单机执行（无验证）。

4. **什么时候需要人工介入？** N 个节点哈希全不一致时，说明模型本身可能有问题，需要人工介入。阈值设在 N=3 比较合理——三个节点说出三种不同结果，那你至少知道"这件事不确定"。
