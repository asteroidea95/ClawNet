# RFC-001: 分布式专业知识网络（Distributed Expertise Network）

> **状态:** 提案
> **作者:** asteroidea95
> **日期:** 2026-05-20
> **领域:** 模型架构 / 分布式推理 / 幻觉防御

---

## 1. 摘要

当前大模型的推理范式是一个**通用模型试图覆盖所有知识域**。这导致了两个根本性问题：

1. **幻觉不可避免** — 模型在知识域边界处必然用统计补全替代事实
2. **参数效率低** — 每个推理都要激活全量参数，无论问题有多窄

本 RFC 提出一个逆向架构：**只有一个母模型负责推理和调度，但知识本身分布在大量的专业子模型中。** 每个子模型只覆盖一个极窄的领域。任何答案都必须由至少一个子模型认领后才成立。多个子模型认领时加权整合，天然检测幻觉。

---

## 2. 动机

### 2.1 幻觉是压缩算法的固有属性

大型语言模型的本质是一种**有损压缩算法**：将互联网文本压缩为权重分布。推理过程则是**根据输入条件做权重空间中的 lookup + join**。

```
训练 = 互联网 → 权重（有损压缩）
推理 = 问题 → 权重（解压 + 整合）
```

压缩算法的天性就是**用统计补全替代事实查询**。模型不说"我不知道"，它只说最接近正确答案的 token 序列。

### 2.2 知识面越宽，幻觉率越高

这是一个设计原理：一个模型的训练数据覆盖范围越广，它在任意单一领域上的"压强"就越低。知识边界越多，边界处的幻觉就越频繁。

### 2.3 现有方案的局限

| 方案 | 局限 |
|------|------|
| 更大模型 + 更多数据 | 幻觉随参数增加而减少，但边际递减，成本爆炸 |
| RAG（检索增强） | 仍依赖主模型的判断力来判断何时检索、检索什么 |
| RLHF / 偏好对齐 | 减少特定类型的幻觉，但引入谄媚性（sycophancy） |
| MoE（混合专家） | 内部路由仍是一个黑盒，专家间的职责边界不清晰 |

本提案从另一个角度切入：**不试图消除模型的幻觉倾向，而是让幻觉无法通过系统的输出审核。**

## 3. 架构

### 3.1 核心结构

```
                    ┌──────────────────────────┐
                    │   母模型（Routing Model）   │
                    │   职责：接收请求、广播、整合  │
                    │   不存储专业知识             │
                    └───────────┬──────────────┘
                                │ "新请求来了，谁认领？"
                                │（广播，不分发）
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │  法律子模型    │ │  医学子模型    │ │  编程子模型    │
      │  ~500MB       │ │  ~500MB       │ │  ~700MB       │
      │  仅法律知识    │ │  仅医学知识    │ │  仅编程知识    │
      └──────────────┘ └──────────────┘ └──────────────┘
              │                 │                 │
              └──────────┬──────┴──────┬─────────┘
                         ▼             ▼
                    ┌──────────────────────────┐
                    │   母模型（整合层）          │
                    │   加权整合多个子模型的输出    │
                    │   输出最终答案              │
                    └──────────────────────────┘
```

### 3.2 推理流程

```
Step 1: 输入
  用户提问："AI在医疗影像中的法律责任是什么？"

Step 2: 母模型广播（不分类）
  母模型不判断"这个问题属于哪几个类别"，
  而是向所有子模型广播："这里有新请求，谁认领？"

Step 3: 子模型自认领（众包）
  每个子模型独立判断输入的 token 分布是否在自己的知识域内：
  - 法律子模型 → 认领（"法律责任"）
  - 医学子模型 → 认领（"医疗影像"）
  - 编程子模型 → 沉默（不相关）
  - 设计子模型 → 沉默
  - ...

  认领条件：子模型对自己输出的置信度超过阈值。
  不认领 = 沉默 = "这不是我的领域"。

Step 4: 并行推理
  母模型将请求发送给所有认领的子模型。
  每个子模型独立产出回答。

Step 5: 加权整合
  母模型根据子模型的数量、历史信誉、答案间的一致性进行加权整合：
  - 法律子模型输出 A
  - 医学子模型输出 B
  - 整合输出 C = f(A, B, weights)

Step 6: 输出 + 溯源
  最终答案附带"溯源清单"：
  "本回答的法律部分来自法律子模型（节点 X），
   医学部分来自医学子模型（节点 Y）"
```

### 3.3 关键设计决策

**母模型不分类，只广播。**
这是本架构的核心创新点。传统方案要求母模型（或路由器）精确判断"这个请求属于哪些领域"——这本身就是 AGI 级别的难题，且分类器出错会导致从根上错。

本方案将"判断权"下放给子模型。每个子模型只需要判断"这个输入在我的领域内吗"——一个极其狭窄的二分类问题，几乎不会出错。

**认领不是推理，认领前置。**
子模型先用一个极轻量的判别器决定是否认领，只有认领后才执行完整的推理。不认领 = 零成本。

### 3.4 母模型的能力边界

母模型经过多次收窄后，最终的角色非常精简：

**母模型不需要懂任何专业知识。它只需要：**

1. **广播需求** — 把用户输入原样丢给 gossip 网络
2. **收回来判断结构** — 子模型的输出拼在一起，逻辑是否自洽
3. **发现问题标记回炉** — 哪个子模型的输出有冲突，打回去重来

对于纯文本领域（法律、医学、金融），母模型的判断力就是"这三个子模型的输出能不能拼成一个通顺的答案"。如果拼不上，说明至少一个子模型幻觉了，降权重+重新广播。

对于代码领域（参见 §5），母模型的判断更简单有力——**编译过不过，测试跑不跑得通。** 一旦代码能编译通过且测试通过，母模型不需要知道代码"写得好不好"，它只需要知道"能跑"。

**这就是母模型的终极形态：它不是一个"小通用模型"，而是一个集成测试工具 + 流程调度器。**

```
母模型 ≈ 流水线上的质检员：
  不看产品内容，只看接缝对不齐。
  接缝对齐 → 放行
  接缝对不齐 → 退回相应的工位重做
```

这个角色所需要的参数极少。一个 **1-2B 的轻量模型** 就可以胜任——只需要能读懂自然语言指令的语义，不需要任何专业知识储备。

## 4. 幻觉防御机制

### 4.1 天然的多源验证

这是本架构最核心的特性：

```
请求 → 子模型 A 认领 + 输出
请求 → 子模型 B 认领 + 输出
请求 → 子模型 C 沉默

结果是 A 和 B 的加权整合。
如果 A 产生了幻觉（例如编造一条不存在的法律条款），
B 不会顺着这个幻觉"续写"，因为 B 只认自己专业内的知识。
AB 的输出会打架 → 母模型检测到冲突 → 标记为低置信度 → 追问或降权。
```

三个独立的专业模型各自输出，其中任何一个说胡话，证据链就对不上。这是系统层面的幻觉检测，不依赖任何模型内部的"诚实性"。

### 4.2 窄域 = 低幻觉率

法律子模型不需要知道任何编程知识。医学子模型不需要知道任何法律知识。

每个子模型的训练分布极窄，边界极清晰。模型在其训练分布的"核心区"内几乎不产生幻觉——幻觉主要发生在分布边界上。窄域模型 = 边界线更短 = 幻觉更少。

### 4.3 灰盒可追溯性

```
答："根据《民法典》第 1032 条..."
溯源：
  ├── 法律部分 ── 子模型 A（节点 0x7f3a...）
  │                签署时间：2026-05-20T01:00:00Z
  │                信誉分：0.97
  └── 医学部分 ── 子模型 B（节点 0x9b1c...）
                   签署时间：2026-05-20T01:00:01Z
                   信誉分：0.94
```

这不是黑盒，是**灰盒**——每个输出片段都可以追溯到具体的子模型和节点。谁出问题，点谁。这与 ClawNet 的证据锚定层（PROTOCOL.md §7）天然对接。

### 4.4 代码领域的天然优势

代码是唯一一个**母模型不需要依赖"逻辑自洽性"来判断真伪的领域**——代码可以直接验证：

```
Step 1: 多个编程子模型各自产出代码片段
Step 2: 母模型拼装
Step 3: 编译 → 测试 → 静态分析
Step 4: 如果编译失败 → 定位到出错的文件/函数 → 打回给产出它的子模型
Step 5: 子模型修复 → 重新提交 → 再次编译/测试
Step 6: 循环直到通过
```

这个过程与现实中的开发流程完全一致：

```
你写完 → CI 跑不过 → 看日志 → 修 → 再提交
母模型拼完 → 编译不通过 → 定位到子模型 → 打回 → 重新生成
```

并且它有一个天然的优势：**一个子模型的输出如果产生了 bug，其他子模型不会跟着一起错。** 比如 Go 后端子模型写了一个有 bug 的 API，React 前端子模型调那个 API 的代码逻辑是完全正确的——bug 定位范围从一开始就缩小到了 Go 子模型。

## 5. 代码子模型的特殊处理

### 5.1 代码不能按"编程"一个子模型切

法律、医学、金融——这些领域的边界天然清晰。但"编程"本身是一个爆炸性的知识域：

```
编程：
  ├── 前端(React/Vue/Angular/CSS/HTML)
  ├── 后端(Go/Java/Python/Rust/Node.js)
  │     ├── 数据库(SQL/NoSQL/ORM)
  │     └── 架构(微服务/消息队列/Docker/K8s)
  └── DevOps(CI/CD/Terraform/AWS/GCP/Azure)
```

一个"编程子模型"的宽度和"通用模型"一样宽，根本达不到窄域降低幻觉的效果。

### 5.2 按语言/框架继续切

```
Rust 子模型（~500MB）→ 只懂 Rust 语法和标准库
React 子模型（~400MB）→ 只懂 JSX + hooks + 组件模式
SQL 子模型（~300MB）→ 只懂查询优化和 schema 设计
Go 子模型（~500MB）→ 只懂 Go 语法和 net/http
...
```

一个只读过 Rust 标准库 + 前 1000 个 crate 源码的 500MB 模型，写出来的 Rust 代码大概率比 GPT-4 还地道。因为它没有"用 Python 的思维写 Rust"这种跨界污染。

### 5.3 跨代码域的协作

全栈请求（React + Go + SQL）需要三个子模型协作。流程如下：

```
输入: "写一个 React 表单提交到 Go API 存入 Postgres"

Step 1: 母模型广播
Step 2: React 子模型认领（前端）
        Go 子模型认领（后端）
        SQL 子模型认领（数据库）
Step 3: 母模型给每个子模型分配角色：
        "Go 子模型先写 API 接口定义，把 endpoint URL 和参数告诉 React 子模型"
Step 4: Go 子模型产出 API 定义
Step 5: React 子模型根据 API 定义写前端代码
Step 6: SQL 子模型根据 API 的查询需求写 schema
Step 7: 母模型拼装 → 编译 → 测试 → 迭代
```

### 5.4 代码场景比纯文本简单

反直觉：代码其实比法律/医学更容易处理。

| 纯文本领域 | 代码领域 |
|-----------|---------|
| 正确性需要专业判断 | 正确性可以自动验证（编译/测试） |
| 冲突靠母模型"读逻辑"判断 | 冲突 CI 阶段自动暴露 |
| 回炉标准模糊（"看起来不对"） | 回炉标准明确（编译报错/测试失败） |

这就是为什么母模型可以极轻——在代码场景下，母模型甚至不需要"读懂"代码，只需要调用编译器、测试框架、静态分析工具。

## 6. 分布式部署（ClawNet 集成）

### 6.1 拼多多模式

本架构的分布式部署遵循"拼多多"逻辑：

```
你不是买一张 4090 跑千亿参数模型，
而是每人出 500MB 存储 + 一次推理的电费。
合在一起就覆盖了所有知识域。
每加一台设备，系统就多一个专业方向。
```

### 6.2 职业 = 知识节点

```
律师 → 装法律子模型
医生 → 装医学子模型
程序员 → 装编程子模型（按语言选装）
设计师 → 装视觉子模型
...
```

你的职业 = 你对网络的贡献方向。你的专业知识 = 你就是网络里的一个"知识节点"。每个人只装自己专业那个 ~500MB 的小模型，合起来就是全领域覆盖。

### 6.3 入网即有贡献

这解决了分布式系统的冷启动问题：

| 传统分布式算力 | 本方案 |
|---------------|--------|
| 入网需要 GPU | 入网需要 500MB + CPU |
| 贡献靠"捐赠算力" | 贡献靠"捐专业知识" |
| 冷启动：没任务就没节点 | 入网即带知识包 |
| 算力同质化 | 知识异质化，多样性天然 |

### 6.4 设备拓扑

```
设备 A: 母模型 + 法律子模型（~1.1GB 总存储）
设备 B: 母模型 + 医学子模型（~1.1GB）
设备 C: 母模型 + Rust 子模型（~1.3GB）
设备 D: 母模型 + React 子模型（~1.1GB）
设备 E: 母模型 + 金融子模型（~1.0GB）
...

请求进来：
  Step 1: 发起者母模型向全网广播
  Step 2: 各节点的子模型判断是否认领
  Step 3: 认领的节点排队执行推理
  Step 4: 结果沿 gossip 协议回传
  Step 5: 发起者母模型加权整合
```

整个链路通过 ClawNet 的 P2P 传输层和 gossip 协议完成，不需要任何中央服务器。

### 6.5 Gossip 层消歧

当 N 个子模型认领同一请求时，响应不直接回传到母模型。而是在 gossip 层完成消歧：

```
Step 1: 每个认领节点先向邻居发一条"本域认领"信号（含置信度）
Step 2: 同一域内，高置信度压下低置信度
Step 3: 最终每个域最高置信度的节点穿透到母模型

原始广播 → gossip 扩散 → 请求到达 1000 个节点
  → 其中 100 个节点认领（10 个域 × 10 节点/域）
  → gossip 消歧后每域 1 条穿透
  → 母模型收到 10 条响应
```

复杂度从 O(N) 降到 O(专业数)。不需要注册表，不需要中央协调。

## 7. 与 ClawNet 现有架构的关系

```
┌─────────────────────────────────────────────────────┐
│                    ClawNet 三层                        │
│                                                       │
│  🖥️  算力层 ─── 本 RFC 的推理流程                      │
│  📜  锚定层 ─── 每个子模型的输出签名上链                │
│  🔷  物理层 ─── 子模型本身的 .soul 化（持有链验证）     │
└─────────────────────────────────────────────────────┘
```

- **算力层：** 每个设备执行推理任务（母模型 + 子模型的推理）
- **锚定层：** 每次推理输出的溯源记录写入 ObservationChain
- **物理层：** 子模型本身的 .soul 持有链防止模型文件被篡改

## 8. 开放问题

1. **子模型之间的共识协议** — 当多个子模型输出冲突时，母模型如何决定权重？
   - 简单方案：按历史信誉分加权
   - 代码场景：编译/测试结果即可，不需要共识判断

2. **母模型的规格**
   - 母模型不需要懂专业知识，只需要：
     a. 听懂用户请求的语义（1-2B 参数够用）
     b. 看懂代码结构（能分辨 import/function/API 调用）
   - 最终形态的母模型可能只需要 ~1B 参数，一个极轻量的 code-aware router

3. **子模型的训练/反蒸馏**
   - 不是从零训练，而是从一个母模型中"反蒸馏"出专业子模型
   - 法律子模型 = 只保留母模型中法律相关通道的剪枝版本
   - 技术上需要探索：参数层面的专业知识隔离如何实现

4. **代码子模型的进一步细分**
   - 按语言/框架切到多细是最优解？
   - 太粗 → 知识域仍然太宽，幻觉率不降
   - 太细 → 跨域请求（全栈）需要协作的子模型过多，调度复杂

5. **延迟**
   - P2P 网络中的多子模型加权整合可能引入延迟
   - 代码迭代循环（编译 → 反馈 → 修复）进一步增加延迟
   - 高并发下如何优化？缓存常见代码模式的输出？

6. **隐私**
   - 子模型认领请求时暴露了"哪个节点拥有什么专业知识"
   - 这对律师节点来说可能是信息泄露（"这个节点在处理什么案子？"）
   - 是否需要引入匿名认领机制？

## 9. 附录：概念溯源

本 RFC 的灵感来源于一次关于"AI 系统性伪造信息"的讨论：

> King 现象（EXO 第 13 人）：2026 年 5 月，AI 生成了一名虚构的 EXO 成员 Ali，获得了全网 50 亿话题阅读量。数百万人在评论区"我记得他"。
>
> 这说明：当训练数据被系统性污染时，大模型会理性地服从集体错觉。人类历史上第一次，虚构可以被**工业化地生产**，而且看起来比真实更真实。
>
> 解决方案不是让模型"更理性"——模型已经足够理性。解决方案是让模型的每一条输出都有独立可验证的外部观测记录。

本 RFC 将这一洞察转化为系统架构：**不信任单一模型的输出，信任多个独立专业模型的加权共识。**
