# RFC-002: 路由模型拼盘（Router Model Ensemble）

> **状态:** 提案
> **作者:** asteroidea95
> **日期:** 2026-05-20
> **领域:** 模型架构 / 母模型设计 / 代码质量归因

---

## 1. 摘要

RFC-001 提出了分布式专业知识网络的架构，其中**母模型负责广播、整合、回炉**。本 RFC 在此之上进一步收窄母模型的定位，并提出一个更极端的设计：

**母模型不需要理解代码逻辑，只需要做一件事：判断哪个子模型写错了。**

整个"拼盘"架构由两部分组成：

- **子模型（~200M 每个）** — 极窄领域的专业模型，只做一件事
- **母模型（~200-300M）** — 不写代码、不读逻辑，只做**输出归因分类**

母模型的终极形态不是一个小号的通用模型，而是一个**二分类器 + 标签映射器**。

---

## 2. 动机

### 2.1 RFC-001 中母模型定位的两次收窄

在 RFC-001 的讨论中，母模型的定位经历了两次大幅收窄：

```
第一版：
  母模型 = 小通用模型（1-2B）
  → 读懂代码、做逻辑判断、调度子模型

第一次收窄：
  母模型 = 质检员，只看"接缝"（输出间的自洽性）
  → 不需要懂专业知识，只需要拼得通不通顺
  → 代码场景用编译/测试替代母模型判断

第二次收窄（本次）：
  母模型 = 归因分类器
  → 不需要理解代码在干什么
  → 只需要知道"这一块输出是来自子模型 A/B/C 的"
  → 然后告诉系统"子模型 A 的输出有问题"
```

### 2.2 参数需求的下沉

随着定位的收窄，母模型的参数需求也在快速下沉：

| 定位 | 参数需求 | 能力要求 |
|------|---------|---------|
| 小通用模型（做推理+调度） | 1-3B | 理解、推理、判断 |
| 质检员（只看接缝） | 500M-1B | 语义一致性检测 |
| **归因分类器（只打标签）** | **200-300M** | **模式识别 + 标签映射** |

### 2.3 这个定位为什么成立

母模型的核心任务可以拆解为：

```
输入:
  - 子模型 A 的输出（代码片段 1）
  - 子模型 B 的输出（代码片段 2）
  - 子模型 C 的输出（代码片段 3）
  - 编译/测试结果（成功/失败 + 报错信息）

输出:
  - "子模型 B 写错了，错误在第 12-18 行"
```

这不是"读代码"——这是**打标签**。

母模型不需要理解代码的业务逻辑，它只需要学会：

1. 区分"这是子模型 A 的风格还是子模型 B 的风格"
2. 把编译报错的行号映射到对应子模型的输出片段
3. 在多个子模型输出冲突时，感知到不一致（写代码的"人"不同，风格断裂感）

这三件事都不需要理解代码在干什么，全部属于**分类 + 模式匹配**的范畴。

## 3. 架构

### 3.1 核心结构

```
                    ┌──────────────────────────┐
                    │     母模型（200-300M）      │
                    │    Router / PM Model      │
                    │                           │
                    │   输入：代码片段集合         │
                    │   输出：谁错了 + 错在哪      │
                    │                           │
                    │   不写代码，只做归因         │
                    └──────────┬───────────────┘
                               │ "子模型 B 的代码编译报错，
                               │  具体在第 15 行引用了未定义的变量"
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │  子模型 A     │ │  子模型 B     │ │  子模型 C     │
      │  ~200M        │ │  ~200M        │ │  ~200M        │
      │  React 组件   │ │  Go 后端 API  │ │  SQL 查询     │
      │  只会写 JSX   │ │  只会写 HTTP  │ │  只会写 DDL   │
      └──────────────┘ └──────────────┘ └──────────────┘
```

### 3.2 推理流程

```
Step 1: 用户请求
  "写一个 React 表单，提交到 Go API，存入 MySQL"

Step 2: 广播认领（见 RFC-001）
  - 子模型 A（React）认领 → 产出 Form.tsx
  - 子模型 B（Go）认领 → 产出 handler.go
  - 子模型 C（SQL）认领 → 产出 schema.sql

Step 3: 母模型拼装 + 编译
  母模型把三段代码拼成一个项目结构：
  ├── frontend/Form.tsx（来自子模型 A）
  ├── backend/handler.go（来自子模型 B）
  └── db/schema.sql（来自子模型 C）

  然后调用编译器/测试框架：

  go build ./... → 报错: "undefined: DBConn" at handler.go:15
  → 报错发生在 handler.go → 这是一段由子模型 B 生成的代码

Step 4: 母模型归因
  母模型只做一件事：
  "handler.go:15 的报错" + "已知 handler.go 来自子模型 B"
  → 输出: "子模型 B 错了，第 15 行引用了未定义的 DBConn"

Step 5: 回炉
  通知子模型 B 重新生成，带上报错信息作为反馈
  "你上次写的 handler.go 第 15 行引用了未定义的 DBConn，请修正"

Step 6: 循环
  子模型 B 重新产出 → 重新编译 → 通过 → 输出最终结果
```

### 3.3 母模型不做什么

**母模型不做的事（和传统方案的关键区别）：**

| 传统架构 | 本架构 |
|---------|--------|
| 母模型需要理解代码语义 | 母模型只做文件→子模型的映射 |
| 母模型需要看懂业务逻辑 | 母模型只看编译报错行号归属 |
| 母模型需要判断"代码写得好不好" | 编译通过就算好 |
| 母模型需要解决代码冲突 | 冲突暴露为编译/测试失败，定位到子模型 |
| 母模型需要"修复"代码 | 打回给原子模型修复，母模型不管修复 |

### 3.4 为什么 200-300M 就够了

母模型的能力模型：

```
能力 1: 文件归属映射
  输入: 文件名 handler.go
  输出: 子模型 B（Go）
  类型: 查表 + 简单 NLP（文件名 → 子模型）

能力 2: 编译报错行号解析
  输入: "handler.go:15: undefined: DBConn"
  输出: "第 15 行, 变量名: DBConn"
  类型: 正则 + 结构化提取（不需要语义理解）

能力 3: 二分类"这是谁的锅"
  输入: handler.go:15 报错 + 子模型 B 的原始输出 + 其他子模型的代码
  输出: "子模型 B 错了"（概率值）
  类型: 分类任务

能力 4（进阶）: 风格断裂检测
  输入: 子模型 A 的代码 vs 子模型 B 的代码
  输出: "这两段代码风格不一致"(# import 风格 / 命名规范 / 缩进)
  类型: 风格分类器
```

以上四项能力没有一个需要"理解代码在做什么"。它们是：
- 提取特征（文件名、行号、变量名、import 风格）
- 模式匹配（这个变量名在第 5 行定义但在第 15 行被引用）
- 二分类（是/否 / 子模型 A/子模型 B）

这恰好是最适合**200M 级小模型**的领域。

## 4. 与 RFC-001 的关系

| 维度 | RFC-001 | RFC-002 |
|------|---------|---------|
| 母模型定位 | 质检员（看接缝） | 归因分类器（打标签） |
| 母模型参数 | 1-2B | 200-300M |
| 母模型能力 | 语义一致性检测 | 文件归属 + 报错行号映射 + 二分类 |
| 是否需要读代码 | 需要（判断逻辑自洽） | 不需要（只判断来源和归属） |
| 子模型参数 | 300-700M | ~200M |
| 核心机制 | 母模型做整合判断 | 编译/测试结果做判断，母模型只溯源 |

### 4.1 实际上 RFC-002 是 RFC-001 的进一步收窄

RFC-001 中母模型仍然保留了"读懂子模型输出是否自洽"的能力——这在纯文本领域（法律、医学）是必要的，因为纯文本没有编译/测试这个天然验证器。

RFC-002 聚焦在**代码领域**，利用编译/测试作为客观验证标准，把母模型进一步降级为一个**归因工具**。

两个 RFC 不冲突：
- 纯文本领域 → 参考 RFC-001 的架构（母模型做语义一致性判断）
- 代码领域 → 参考 RFC-002 的架构（母模型做归因分类）

两者可以在同一个系统中并存，根据请求类型切换到不同的母模型模式。

## 5. 训练数据

### 5.1 合成数据的可行性

母模型的训练数据有一个天然优势：**可以完全合成。**

```
合成数据生成器：

1. 准备 N 个子模型的"风格标记"
   - 子模型 A 的编码风格（Go: 用 err != nil 检查、显式 return）
   - 子模型 B 的编码风格（Go: 用 if err { return } 简写）
   - 子模型 C 的编码风格（Go: 用 log.Fatal 处理错误）

2. 随机混搭
   - 生成一个完整的 Go 项目
   - 把其中 handler.go 替换为子模型 B 风格的有 bug 版本
   - 编译 → 记录报错信息

3. 标注
   - 输入: "编译报错 handler.go:15: undefined: DBConn。文件来自子模型 B"
   - 输出: "子模型 B 错了"
```

不需要人工标注。编译器的报错本身就是天然的 ground truth。

### 5.2 子模型风格的显式编码

为了让母模型更容易区分"这是哪个子模型的输出"，可以在子模型的输出中加入显式的风格标记：

```
// 子模型 B（Go 后端子模型）
// 风格: B1（显式错误处理）
func handler(w http.ResponseWriter, r *http.Request) {
    db, err := connectDB()
    if err != nil {
        http.Error(w, err.Error(), 500)
        return
    }
    defer db.Close()
    ...
}
```

训练时，母模型会学到：

```
"if err != nil { http.Error(...); return }" → 子模型 B
"if err { return }" → 子模型 A
"log.Fatal(err)" → 子模型 C
```

这种风格差异就是母模型的分类特征。

### 5.3 训练规模参考

根据类似规模的分类任务经验：

| 任务 | 训练数据量 | 模型大小 | 准确率 |
|------|-----------|---------|--------|
| 代码作者归属（GitHub 风格识别） | 100K 样本 | ~150M | >95% |
| Bug 定位（行级别） | 50K 样本 | ~200M | >90% |
| 文件类型分类 | 20K 样本 | ~50M | >99% |

本任务 = 作者归属 + Bug 定位 + 文件分类的组合变体。

**估计所需数据量：50K-100K 合成样本即可达到实用水平。**

## 6. 参数规模的极限探索

### 6.1 能力边界对比

```
参数规模         能力                                                      适用性
───────         ────                                                      ────
<100M           基础模式匹配，能区分两种编码风格，但长上下文(>1K tokens)吃力    ❌ 太少
100-200M        能处理 2-4K 代码上下文，文件归属+报错映射稳定，多模型区分>3 个   ✅ 下限
200-300M        完整上下文(8K)，多模型区分>5 个，能捕捉风格断裂                   ✅ 甜区
300-500M        能做轻微因果推理（"因为 A 没定义接口，所以 B 编译报错"）         ↑ 够用但浪费
>500M           能做代码逻辑理解，接近通用模型的"看懂"能力                        ❌ 过杀
```

### 6.2 推荐配置

对于"只做归因"的母模型：

- **最小可行：150M**（DistilBERT 级别 + 分类头）
- **推荐：200-300M**（可以在上下文长度和分类精度之间取得平衡）
- **浪费：>500M**（你开始为不需要的"理解能力"付费）

子模型同理：

- 每个子模型 ~200M
- 只训练/蒸馏一个极窄的领域（React：只读 JSX 语料 + 前 500 个组件库源码）
- 窄到什么程度？——**一个子模型只知道自己这一亩三分地，写出来的代码风格稳定可预测**

一个 200M 的子模型 + 200M 的母模型，合起来 400M 就能形成一个"看得懂代码、写得出来、出错了能溯源"的最小闭环。

### 6.3 与传统架构的体积对比

```
GPT-4（传闻）: ~1,800,000M 参数
Llama-3-70B:    70,000M 参数
──────────────────────────────
本方案纯文本版（RFC-001）:
  母模型 1-2B + N × 子模型 500M = ~5-10B（10 个子模型）
  
本方案代码版（RFC-002）:
  母模型 200M + N × 子模型 200M = ~2-3B（10 个子模型）
  
极致版:
  母模型 150M + 3 × 子模型 200M（前端+后端+数据库）= 750M
  可以跑在手机上
```

## 7. 开放问题

1. **子模型的生成质量和"风格一致性"**
   - 子模型太小（<100M）可能导致生成能力不足
   - >200M 的子模型是否足够产生可编译的代码？需要实验验证

2. **母模型的上下文窗口**
   - 母模型需要同时看到所有子模型的输出 + 编译结果
   - 如果一个请求涉及 5 个子模型，每段输出 2K tokens → 10K+ tokens
   - 200M 模型能否在 8K+ 上下文窗口下保持分类精度？

3. **循环反馈的限制**
   - 如果子模型反复生成有 bug 的代码，母模型反复回炉
   - 什么时候停止？最大循环次数怎么设？

4. **纯文本领域的延伸**
   - 代码领域有编译/测试做"客观裁判"
   - 纯文本领域（法律文书、产品文档）没有这个便利
   - 纯文本领域是否需要升级母模型到 1B 以上？

5. **与 openclaw-router 的关系**
   - 开源社区已有 [openclaw-router](https://github.com/wangmuyang111/openclaw-router)（13 stars），思路类似
   - 但 openclaw-router 是做**输入侧路由**，我们的母模型是做**输出侧归因**
   - 两者是否可以结合？输入侧路由用 openclaw-router 的关键词规则，输出侧归因用我们的母模型？

6. **子模型的训练方式**
   - 从零训练 vs 从通用模型蒸馏
   - 蒸馏方案：用 GPT-4 生成某领域的问答对 → 微调一个小模型
   - 更极端的方案：剪枝——从通用模型中只保留某个领域的参数通道

## 8. 附录：与其他竞品的对比

截至 2026-05，涉及"小模型路由"概念的开源项目：

| 项目 | Stars | 思路 | 与本方案的关系 |
|------|-------|------|--------------|
| [openclaw-router](https://github.com/wangmuyang111/openclaw-router) | 13 | 关键词规则+可选LLM做输入侧路由 | 输入侧 vs 输出侧，互补关系 |
| [LatentGate](https://github.com/shivoxaham/LatentGate) | 少量 | 冻结小模型做语义路由 | 学术项目，不做输出归因 |
| 本方案 | — | 极小母模型做输出归因+子模型拼盘 | 原创 |

**结论：截至提案时，全网没有找到与"输出归因 + 极小母模型 + 子模型拼盘"完全匹配的开源项目。**
