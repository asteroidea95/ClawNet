> 草稿阶段，会随开发迭代。

## 1. 节点发现协议

基于 SWIM 的 gossip 成员管理。

### 1.1 节点标识

```proto
message NodeInfo {
    bytes  node_id = 1;          // Ed25519 公钥的哈希
    string host = 2;             // 公网 IP 或局域网地址
    uint32 port = 3;             // 监听端口
    double uptime_hours = 4;     // 累计在线时长
    uint32 token_balance = 5;    // 令牌余额（公告用近似值）
    repeated string caps = 6;    // 能力标签 ["dom", "gpu", "video-gen", ...]
}
```

### 1.2 成员协议

```
PING(seq)                     → ACK(seq)
PING-REQ(target, seq)         → ACK(seq) 或 NACK
INDIRECT-PING(target, seq)    → 由邻居转发探测
ANNOUNCE(node_info)           → 新节点 / 状态变更广播
```

### 1.3 故障检测

- 每个节点维持一个**怀疑列表**
- 节点在 K 轮 gossip 周期内无响应 → 标记为 `SUSPECT`
- 向随机邻居发起 `PING-REQ` 间接探测
- 间接探测失败 → 标记 `DEAD` → 全网广播

## 2. CRDT 令牌台账

使用 **G-Counter** (Grow-only Counter) 和 **AWORSet** (Add-Wins Observed-Removed Set) 的组合：

### 2.1 数据结构

```rust
struct TokenEntry {
    from: NodeId,          // 支付方
    to: NodeId,            // 接收方
    amount: u32,           // 令牌数
    signature: [u8; 64],   // 支付方的 Ed25519 签名
    timestamp: u64,        // 本地时间戳
    seq: u64,              // 单调递增序列号
}

// 每个节点维护：
type TokenLedger = Map<(NodeId, NodeId), GCounter>
// 合并规则：取各副本对应计数器的最大值
```

### 2.2 交易流程

```
1. 贡献者完成任务后，向发起者发送结果
2. 发起者验证结果 → 生成 TokenEntry
3. 发起者签名 TokenEntry → 发送给贡献者
4. 双方各自添加到本地台账
5. 台账随 gossip 同步到其他节点
```

### 2.3 冲突解决

CRDT 保证无冲突合并：
- G-Counter：两个副本合并时取每个元素的最大值
- AWORSet：元素同时被添加和删除时，添加胜出

## 3. 任务公告协议

### 3.1 任务公告

```rust
struct TaskAnnouncement {
    task_id: [u8; 32],        // 哈希
    task_type: TaskType,      // dom | image-gen | video-gen | inference
    description: String,      // 人类可读描述
    requirements: Requirements, // 硬件/能力要求
    reward_tokens: u32,       // 支付的令牌数
    deadline: u64,            // 截止时间戳
    initiator: NodeId,        // 发起者
    signature: [u8; 64],      // 发起者签名
}
```

### 3.2 任务认领

```rust
struct TaskClaim {
    task_id: [u8; 32],
    claimer: NodeId,
    capability_proof: String,  // 能力证明
}
```

### 3.3 状态机

```
CREATED → ASSIGNED → IN_PROGRESS → COMPLETED
    |          |            |
    v          v            v
EXPIRED   EXPIRED      FAILED
```

## 4. 传输层

### 4.1 小消息（gossip 通道）

大小 < 64KB，直接嵌入 gossip 消息体：

- 节点心跳
- 任务公告
- 令牌交易
- 任务认领

### 4.2 大文件（直连通道）

大小 > 64KB，通过直连传输：

- 视频帧序列
- 模型分片
- 批量搜索结果

直连协议：支持 libp2p 或 原生 TCP + STUN/TURN。

## 6. 安全模型

### 6.1 通信安全

- 所有节点间通信使用 Ed25519 签名
- 可选 TLS 加密传输

### 6.2 Sybil 攻击防御

ClawNet 不预防 Sybil 攻击。因为令牌系统的经济模型使得**攻击的成本高于收益**：
- 新节点令牌余额为 0，优先级最低
- 大量创建假节点需要大量贡献真实算力才能获得令牌
- 恶意行为被举报后，该节点 ID 的信任积分清零

### 6.3 结果可用性

- 任务发起者同时将任务发给多个节点 → 冗余执行
- 至少取前 K 个返回结果
- 避免单点故障导致的等待

## 7. 证据锚定协议 (Evidence Anchoring)

> 每条任务执行记录同时是一个**分布式证据锚点**。
> ClawNet 的台账不仅是令牌账本，也是网络对事实的集体观测日志。

### 7.1 动机

在数字信息可被 AI 系统性伪造的时代（文本、图像、视频、语音均可生成），
判断一条信息的来源是否可靠，需要知道**在它声称产生的时间点，是否有
可信的观测者见证了这一事件**。

ClawNet 的分布式节点网络天然具备这一能力——每一笔计算都是
一个观测事件，观测结果被签名并传播到全网，形成不可篡改的
**数字事实锚点**。

### 7.2 观测日志

每个节点执行任务时，自动生成观测日志记录：

```rust
struct Observation {
    node_id: NodeId,            // 观测者
    task_type: TaskType,        // 任务类型
    task_hash: Hash,            // 任务输入哈希
    observed_data: Hash,        // 观测结果哈希
    source_ref: Option<String>, // 来源引用（URL、页面标题等）
    timestamp: u64,            // 观测发生时间
    prev_observation: Hash,    // 本节点上一条观测的哈希（形成个人链）
    signature: Signature,      // 节点签名
}
```

记录中包含的 `prev_observation` 字段将同一节点产生的前后观测
链接成一条不可篡改的个人时间链。修改任何历史观测都会导致后续所有
观测的哈希链断裂。

### 7.3 证据锚定流程

```
1. 节点被分配一个任务（如：去某个URL检索关键词）
2. 执行前记录任务输入 hash + 时间戳 → 获得任务锚点
3. 执行后记录结果 hash + 来源引用 → 获得结果锚点
4. 两条锚点用 prev_observation 链接到本节点的观测链上
5. 锚点随 gossip 协议广播到全网
6. 其他节点验证签名的有效性后，存入分布式台账
```

### 7.4 负存在证明

> 如果某信息的声称时间点，全网没有节点观测到它——
> 那它的来源就存疑。

这一原理是 ClawNet 证据锚定区别于传统事实核查的核心。
不依赖某个中央权威判定真伪，而是依赖网络的**集体观测覆盖**，
而网络的规模越大，负存在证明的说服力越强。

### 7.5 信任评分

每个节点积累基于以下维度的信任评分：

| 维度 | 计算方式 | 权重 |
|---|---|---|
| 观测时长 | 首次观测至今的日历天数 | 中 |
| 观测密度 | 总观测数 / 活跃天数 | 中 |
| 交叉验证率 | 本节点观测被其他节点验证一致的次数 | 高 |
| 异常率 | 被其他节点报告结果异常的次数（反向） | 高 |
| 令牌净值 | 净令牌余额反映的贡献历史 | 低 |

信任评分影响：
- **证据权重**：在负存在证明查询中，高信任节点的
观测结果被赋予更高权重
- **观察者保证金**：高信任节点在作伪证时损失更大（声誉+令牌）
- **新节点冷启动**：新节点需要积累初始观测并通过交叉验证才能获得权重

### 7.6 作伪证的经济惩罚

节点的观测记录如果被发现是伪造的（被多个独立节点交叉验证否决）：

```
第1次违规 → 信任分归零，保留令牌
第2次违规 → 信任分归零 + 扣除净令牌的 50%
第3次违规 → 全局公告恶意节点 ID，所有节点拒绝其连接
```

惩罚的设计原则是：**作伪证的预期收益必须远低于诚实观测的预期收益**。
由于 ClawNet 的令牌系统不涉及货币兑换，伪证者无法"套现离场"，
恶意行为的唯一获利是操控历史的叙事权——而一旦节点 ID 被标记为恶意，
其所有历史观测记录也一并被标记为不可信，从而抵消其操控效果。

### 7.7 与区块链的关系

ClawNet 的证据锚定层不是区块链，但具备区块链的核心优势，
同时避免了区块链的典型缺陷：

| | 区块链 | ClawNet 证据锚定 |
|---|---|---|
| 共识机制 | PoW / PoS，高能耗 | CRDT 最终一致，零额外能耗 |
| 写入资格 | 任何人（需消耗 Gas/Gas 无需） | 仅任务执行者 |
| 存储 | 全节点存全部历史 | 分布式台账，节点按需存储 |
| 可扩展性 | 受限于区块大小和 TPS | 随节点数线性扩展 |
| 抗篡改 | 51% 攻击 | 需要控制网络中信任权重 > 50% 的节点 |
| 经济激励 | 加密货币 | 令牌（网络优先使用权，非货币） |
| 适用场景 | 金融交易 | 事实记录与计算证明 |

### 7.8 局限

- **未覆盖的时间盲区**：如果某个时间点 ClawNet 的网络规模较小或没有
  活跃节点，负存在证明无效（一个不存在于网络观测中的事件也可能是真实的）
- **信任集中的风险**：如果少数节点控制了网络大部分信任权重，
  这些节点可以合谋篡改观测记录——但代价是自己的声誉
- **观测不等于理解**：节点记录了"某个页面在某个时间点的内容"，
  但不验证这个内容本身的真实性（这需要任务本身的正确设计）
- **隐私**：观测日志记录了每个节点执行了什么任务、访问了哪些 URL，
  这会暴露用户的兴趣和行为模式。后续版本需要引入匿名观测机制
  （如差分隐私或环签名）
